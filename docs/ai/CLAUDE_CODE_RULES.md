# Правила для Claude Code

> Это **операционные** правила (как именно работать в репо). Поведенческие принципы и стек — в [`../../CLAUDE.md`](../../CLAUDE.md). Этот файл — короче и оперативнее.

## Перед каждой сессией

1. Прочитать [`../../STATE.md`](../../STATE.md) → понять текущий контекст.
2. Прочитать последние 10–20 строк [`../../docs/WORK_LOG.md`](../WORK_LOG.md) (если ведётся).
3. Сверить ветку: фичи → `feature/…-NNN`, фикс → `fix/…-NNN`, инфраструктура → `chore/…`.
4. Не работать в `main` напрямую.

## В процессе

- **Один шаг = один коммит.** Большие правки бить на серию коммитов с понятным scope.
- **Conventional Commits обязательны.** `feat(scope): …`, `fix(scope): …`, `chore(scope): …`, `docs(scope): …`, `test(scope): …`, `refactor(scope): …`.
- **Документация в том же PR.** Если меняешь поведение — обнови README/CLAUDE/STATE/CHANGELOG.
- **Серверные действия не делать.** Любая задача с `kubectl`, `gh api`, изменение прав, выдача токенов — issue с тегом `[devops:vika]`.
- **Секреты не коммитить.** При случайном попадании — немедленно остановиться и создать P0-issue для Вики на ротацию.
- **Существующие конвенции уважать.** Не «улучшать» соседний код без явного запроса.

## После сессии

1. Обновить [`../../STATE.md`](../../STATE.md): «В работе сейчас» → что сделано / что осталось / что заблокировано.
2. Если принято архитектурное решение — добавить запись в [`../../DECISIONS.md`](../../DECISIONS.md). Большое — отдельный ADR в `docs/adr/`.
3. Если ввёл/изменил публичный API — обновить `docs/api/` и `CHANGELOG.md`.
4. Push в свою ветку. PR — только когда задача целостно завершена.

## Чего не делать

- Не пушить force в `main` или общие ветки.
- Не использовать `--no-verify`, `--no-gpg-sign` без явной просьбы пользователя.
- Не удалять файлы без подтверждения — перемещать в `_archive/YYYY-MM-DD/`.
- Не создавать «спекулятивные» абстракции под несуществующие требования (см. CLAUDE.md, принцип «Простота прежде всего»).
- Не молча выбирать одну из нескольких интерпретаций — спросить.
- Не создавать `.md` без явного запроса (кроме случаев, когда `REPO_SETUP_PROMPT.md` требует).

## Когда останавливаться и спрашивать

- Найдены секреты в git history.
- Задача требует изменения схемы БД с потенциальной потерей данных.
- Задача требует удаления файлов/директорий, не созданных текущей сессией.
- Несколько разумных интерпретаций требования — выбрать молча нельзя.
- Стоимость операции (LLM-вызовы, S3 traffic, …) превышает «дешёвый» уровень — подтвердить.
